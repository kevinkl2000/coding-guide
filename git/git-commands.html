Common Git Commands

From day-one commands to production branch workflow. Each block includes what it does, when to use it, and what to watch out for.

How To Use This Page

  • Run commands in order when shown as a sequence.
  • Replace ticket names, versions, and branch names with your own values.
  • Use git status before and after major commands so you always know your state.
STATUS

Check current state

Use this before editing or merging. It shows changed files, recent history, and which branches exist locally.

git status
git log --oneline -10
git branch
COMMIT

Stage and commit

git add . stages all changed files in the current folder tree. git commit records a snapshot with a message that explains intent.

git add .
git commit -m "feat: add profile settings page"
UNDO

Unstage files

git restore --staged removes files from staging but keeps edits. git restore path/to/file resets working changes for one file to the last commit.

git restore --staged .
git restore path/to/file
BRANCH

Create ticket branch

Always branch from an updated main if your team policy says ticket branches start from main. This keeps each ticket isolated and reviewable.

git switch main
git pull origin main
git switch -c ticket/789-user-avatar-upload
SYNC

Update from remote

git fetch downloads remote updates without changing your files. git pull fetches plus merges the chosen branch into your current branch.

git fetch origin
git switch dev
git pull origin dev
MERGE

Merge ticket into dev locally

--no-ff forces a merge commit so integration points stay visible in history. This is useful for auditing what entered dev and when.

git switch dev
git pull origin dev
git merge --no-ff ticket/789-user-avatar-upload
git push origin dev
FLOW

Production team workflow

This sequence models your requested production flow: ticket branches are created from main, integrated into dev, then released from dev to main after QA.

# 1) prep branches
git switch main
git pull origin main
git switch dev
git pull origin dev

# 2) create ticket from main
git switch main
git switch -c ticket/123-short-description

# 3) develop and push
git add .
git commit -m "feat: implement ticket 123"
git push -u origin ticket/123-short-description

# 4) PR ticket -> dev
# 5) after QA, PR dev -> main
RELEASE

Tag release from main

Annotated tags mark release points in history. They let you redeploy or compare exactly what changed between versions.

git switch main
git pull origin main
git tag -a v2.0.0 -m "Release 2.0.0"
git push origin v2.0.0
