Lesson 5

Sessions, APIs, and Security

These topics are where many real PHP applications become useful or risky. Good defaults matter.

Sessions

Sessions let you keep data between requests, which is useful for logins, carts, and multi-step workflows.

<?php
session_start();
$_SESSION["user_id"] = 42;

echo $_SESSION["user_id"];
?>

Output

42

Returning JSON for APIs

<?php
header("Content-Type: application/json");

echo json_encode([
    "status" => "success",
    "message" => "Guide loaded",
    "items" => ["syntax", "forms", "pdo"],
]);
?>

Output

{"status":"success","message":"Guide loaded","items":["syntax","forms","pdo"]}

Password Hashing

Never store plain text passwords. Use password_hash() and password_verify().

<?php
$hash = password_hash($password, PASSWORD_DEFAULT);

if (password_verify($enteredPassword, $hash)) {
    echo "Password is correct.";
}
?>

Output

Password is correct.

Common Security Rules

  • Escape output to prevent XSS.
  • Use prepared statements to prevent SQL injection.
  • Regenerate session IDs after login.
  • Validate uploaded files before storing them.
  • Use CSRF protection for sensitive form actions.

Simple CSRF Token Pattern

<?php
session_start();

if (empty($_SESSION["csrf_token"])) {
    $_SESSION["csrf_token"] = bin2hex(random_bytes(32));
}
?>

<input type="hidden" name="csrf_token" value="<?= $_SESSION["csrf_token"] ?>">

Output

<input type="hidden" name="csrf_token" value="4f8a21c2d20f7bdc1f84f0d17b4b7b61c4b0c3db63f7a8a2f3f4a14b7ea8d261">

Production Mindset

The difference between a demo app and a production app is usually validation, error handling, logging, and security discipline. Build those habits early.

Final Checklist

  • Start sessions before output.
  • Return JSON with the correct content type.
  • Hash passwords instead of encrypting them yourself.
  • Assume every input can be abused until validated.
